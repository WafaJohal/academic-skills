# Common HREC/LR Reviewer Issues

Distilled from common reviewer feedback patterns across multiple ethics applications.
These are the issues that come up most frequently. The writer should proactively avoid them;
the reviewer should explicitly check for each one.

Organised by section of the application and associated documents.

---

## A. PLAIN LANGUAGE SUMMARY (Application Section B)

**A1. "No risks" when inconvenience IS a risk** 🔴 VERY COMMON
The committee treats inconvenience (time, mild fatigue, repeated questions) as a form of risk.
Never write "risks are minimal/none" — always acknowledge inconvenience explicitly.
Fix: "The primary risk to participants is the time required to complete [X]. Participants may
skip any question or withdraw at any time without consequence."

**A2. Summary too long or too technical** 🟠
The plain language summary must be genuinely plain. Use:
- Short paragraphs or dot points
- Active voice ("We will collect..." not "Data will be collected...")
- No unexplained jargon (see A3)

**A3. Technical terms not explained** 🟠 COMMON
Terms like "haptic technology", "cognitive load", "shared autonomy", "reinforcement learning",
"exploratory factor analysis", "teleoperation" must be defined in plain language the first time.
Fix: Include a one-sentence lay definition in parentheses.

**A4. Summary missing required elements** 🟠
The plain language summary must include ALL of: aims, methods, participant numbers,
time commitment per activity, incentives/reimbursement, consent process, risks, benefits.
Missing any of these will generate a revision request.

## B. PLS — WHAT WILL I BE ASKED TO DO?

**B1. Audio/video recording not stated explicitly** 🔴 VERY COMMON
If ANY session will be audio or video recorded, this MUST be stated explicitly in the PLS.
Do NOT write "may be audio and/or video taped" — write "will be audio-recorded".
Also: state who will transcribe the recording and how (e.g., "AI transcription tool reviewed
by the researcher", "professional transcription service under confidentiality agreement").

**B2. Location not specified** 🟠 COMMON
Every activity must state WHERE it will take place.
e.g., "You will attend an in-person session at [name of venue or type of location]"
or "You will complete an online survey via a secure link sent to your email."
Do NOT write "the session will be held at a mutually agreed location."

**B3. Time commitment missing or incomplete** 🟠 COMMON
State time commitment for EACH activity separately.
ALSO include time to read the PLS and consent form (typically 5–10 minutes).
e.g., "Reading this information and signing the consent form (~10 min) + completing the
survey (~20 min) = approximately 30 minutes in total."

**B4. Eye-tracking / gaze data not described** 🟠 (for studies using eye tracking)
If eye-tracking is used, state this explicitly: what is captured (gaze direction, potentially
video of the face), how it will be stored, and whether video of participants' faces is retained.
The committee treats face video as identifiable and requires specific handling details.

**B5. Activities inconsistent across documents** 🔴
The activities listed in the PLS must exactly match the consent form bullet points.
The consent form bullets must exactly match what is described in the application.
Any inconsistency triggers a revision request.

**B6. Numbered dot points recommended for activities** 🟡
The committee prefers numbered dot points for the "What will I be asked to do?" section,
especially when there are multiple sequential activities.

## C. PLS — WHAT WILL HAPPEN TO MY INFORMATION? / DATA MANAGEMENT

**C1. Withdrawal rights inconsistent with data type** 🟠 COMMON
If data is anonymous or de-identified, the right to withdraw DATA (not just participation)
is limited. You cannot "withdraw your data" from an anonymous survey once submitted.
Fix: State clearly: "As your responses are anonymous, it will not be possible to remove
your individual responses once you have submitted the survey."
Do NOT use a standard consent form that promises data withdrawal if it's not feasible.

**C2. Storage: cloud storage platform specifics** 🟡
If using institutional cloud storage, specify the platform explicitly (e.g., OneDrive vs
SharePoint). Confirm access controls are appropriate and institution-approved.

**C3. Transport of data from off-site locations** 🟠
If data is collected at conferences, external sites, or overseas, explicitly state how
recordings/data will be securely transported to approved institutional storage.
e.g., "Recordings will be transferred to institution-approved cloud storage via encrypted
upload immediately after the session. No data will be stored on personal devices beyond
the recording session."

**C4. "Physical data" undefined** 🟡
If you refer to physical/non-digital data (field notes, paper prototypes, artefacts),
define what this includes and how it will be stored and eventually destroyed.

**C5. Reimbursement mechanism for anonymous participants** 🟠
If participants are anonymous (e.g., recruited via mailing lists, not Prolific) but will
receive payment/gift cards, explain the mechanism: how they receive reimbursement
without being identified. e.g., "Participants will be sent a unique code to a gift card
platform after completion, via the survey platform. No name or email is retained by the
research team."

**C6. Currency inconsistency for international participants** 🟠
If recruiting internationally, payment must be in a currency/platform consistent with
participants' location. Prolific handles currency conversion automatically — note this if
using Prolific.

## D. PLS — GENERAL ISSUES

**D1. Technical abbreviations not spelled out on first use** 🟡 COMMON
Abbreviations must be spelled out in full on first use, even common ones.
e.g., "Artificial Intelligence (AI)", "Human-Computer Interaction (HCI)", "Extended Reality (XR)"
This applies to recruitment materials as well as the PLS.

**D2. PLS provided "before and during the interview"** 🟠
Do not write "participants will be informed before and during the interview" for survey-based research.
If the method is an online survey, the PLS is provided online before the survey begins.
State the actual delivery mechanism: "Participants will read this PLS online before beginning the survey."

**D3. PLS too long** 🟡
The committee expects the PLS to be succinct. If the "What is this research about?" section
is more than 2–3 short paragraphs, it should be shortened. Use plain language rather than
reproducing the application's background section.

**D4. Dates and version numbers out of date** 🔴
All participant-facing documents must have current dates and correct version numbers in the footer.
The committee has rejected documents with dates in the past.
Fix: Check footer before every submission and resubmission.

**D5. Confidentiality statement for workshop participants** 🟡
For focus groups and workshops, consider adding a statement asking participants to
keep each other's contributions confidential: "While we will take steps to protect your
confidentiality, we ask that all participants treat discussions within the workshop as
confidential and not share others' responses outside the session."

## E. CONSENT FORM ISSUES

**E1. Audio recording: "may be" vs "will be"** 🔴 COMMON
Consent forms must say "will be audio-recorded" (or video-recorded) — not "may be".
If you are not sure in advance, resolve this before submission. The committee will not
approve conditional recording language.

**E2. Logical order of consent bullets** 🟠
Consent form bullets should follow a logical order:
1. Consent to participate + have received PLS
2. Understand the purpose
3. For research purposes only
4. Effects explained
5. Activities (what they'll do)
6. Recording (if applicable)
7. Voluntary + right to withdraw
8. Data storage + retention period
9. Confidentiality + access control
10. Anonymity caveat (if small sample)
11. Consent form retained by researcher

**E3. Version number, date, Project ID in footer** 🔴
Consent form footers must contain: Project ID Number + Project Start Date + Version.
Missing any of these will generate a query.

**E4. Consent form activities don't match PLS** 🔴
Each bullet in the consent form describing participant activities must match the
corresponding text in the PLS exactly. Committees check these for consistency.

## F. RISK SECTION ISSUES

**F1. Risks to participants: only generic mitigations** 🟠 COMMON
Listing "participants can withdraw at any time" as the only mitigation for every risk
is insufficient. Each identified risk needs a specific mitigation strategy.
e.g., for privacy risk: describe where data is stored, who has access, and how it is protected.

**F2. Risks to researchers not addressed** 🟠
For face-to-face studies, the committee will ask about:
- Physical risks (travel to site, site hazards)
- Psychological risks (exposure to distressing content during interviews)
Even if risk is low, acknowledge it and describe mitigation.

**F3. Anonymity, confidentiality, and reputation risks in workshops** 🟠 COMMON
For workshops, focus groups, or co-design sessions:
- Participants may be identifiable to each other
- Participants may disclose sensitive professional opinions
- Risk to professional reputation if comments leave the room
Must be acknowledged as a risk category with mitigation strategies.

**F4. Risk-benefit justification must match risk identification** 🟠
If the risk assessment section identifies risks, the justification section must address
ALL of them. Committees check for coherence between the two sections.

## G. PARTICIPANT ISSUES

**G1. Age eligibility phrasing** 🟡
Preferred phrasing: "Adults between the ages of 18 and [N]" rather than "18 years or older"
when there is an upper age limit. If no upper limit, "adults aged 18 and over" is acceptable.

**G2. Participant count inconsistency** 🟠
If your study has multiple phases or participant groups (e.g., Delphi experts + survey participants),
be explicit about whether the total count includes ALL groups.
State: "This includes [X] expert participants for Stage 1 and [Y] participants for Stages 2–4."

**G3. Clarify participant task count** 🟠
If participants complete multiple tasks or conditions, state clearly: "Each participant will
complete [one / two / X] tasks." Ambiguity here will generate a query.

**G4. Robotics / technology expertise expectations** 🟡
If using technology (robots, VR/XR, haptic devices) with participants, state the assumed
level of expertise required. If no expertise needed, say so explicitly.

**G5. Speaking on behalf of others / representation concerns** 🟠
If participants are asked to represent a group (e.g., neurodivergent people speaking
about others' needs), the committee may raise concerns about representation.
Address: "Participants are asked to speak from their own experience only. We acknowledge
the diversity within [group] and do not generalise findings beyond our specific sample."

**G6. Student-staff power dynamics** 🟠
If students or staff could participate in a study run by members of their own institution,
address the dependent relationship risk explicitly in Section G (P3) and in the PLS.
State how coercion risk is minimised (e.g., researchers not in supervisory role over participants).

## H. PERSONNEL AND TEAM ISSUES

**H1. External personnel missing organisation details** 🟠 COMMON
Every external collaborator must have: name, title, institution/organisation, department, email.
Do not leave Organisation blank for external personnel.

**H2. Supervisor role when multiple students listed** 🟡
If two student researchers are listed, clarify who the Project Supervisor is supervising and
what each student's responsibilities are. The committee may query whether supervision
arrangements are adequate.

**H3. Project Supervisor ethics committee membership** 🟠
If the Project Supervisor is a current HREC member, this should be acknowledged.
If their membership status has changed, update this in the application.
The committee may ask for confirmation of current status.

**H4. Monitoring and compliance plan — insufficient detail** 🟠
"Regular meetings" is not sufficient. Describe:
- Meeting frequency (e.g., weekly)
- What is reviewed (protocol adherence, data collection procedures)
- How the supervisor approves materials before use
- How issues are escalated

## I. RECRUITMENT MATERIALS ISSUES

**I1. Technical language in recruitment materials** 🟠 COMMON
Recruitment emails and flyers must be in plain language. Technical terms from the
application (e.g., "shared autonomy", "haptic control", "reinforcement learning") must
be explained or replaced with plain language equivalents.

**I2. Typos and language errors** 🟠
Proofread recruitment materials carefully. The committee has flagged specific errors
(e.g., "then" instead of "than", mismatched tenses).

**I3. Dates out of date** 🔴
Recruitment flyers with past dates will be flagged and must be updated before approval.

**I4. Contact information missing** 🟠
Recruitment materials must include a contact email address for participant enquiries.

**I5. Participant numbers and time commitment in flyers** 🟠
Recruitment flyers should state how many participants are being recruited and how long
participation will take, so participants can make an informed decision whether to respond.

## J. APPLICATION-LEVEL ISSUES

**J1. Amendment vs new application** 🔴
If a project involves a substantially new population, new topic, or significant change to
methods, the committee may require a new application rather than an amendment.
Key trigger: new population + new topic together = new application required.

**J2. Location inconsistency** 🟠 COMMON
"Where will participants be located?" and "In what environments?" must be consistent
with the data collection description. If workshops occur at conferences or external venues,
these must appear in the location checkboxes.

**J3. Study materials not provided** 🟠
The committee expects to see examples of all materials participants will interact with:
survey instruments, vignettes, interview guides, workshop plans, recruitment scripts.
For novel items (hypothetical scenarios, vignettes), provide examples.
Committees will request these if not provided.

**J4. Abbreviation in project title** 🟡
Project titles in Infonetica should avoid unexplained acronyms. The full title must be
clear to a non-specialist committee member.

**J5. Environment checklist incomplete** 🟡
If research occurs at a university (even online via institutional platforms), tick "Higher education
(University/TAFE)" in addition to "Online". The committee has noted omissions here.

**J6. "What will happen to information about me" in PLS not matching application** 🔴
The PLS data section must match the application's data storage and retention sections exactly.
If the application mentions multiple storage systems, all must be stated consistently across documents.

**J7. Abbreviations in application body** 🟡
Spell out on first use in the application body, not just participant-facing documents.
e.g., "Artificial Intelligence (AI)", "Large Language Model (LLM)", "Human-Robot Interaction (HRI)"

## QUICK CHECKLIST — HIGH-FREQUENCY ISSUES

Before submitting, confirm these are all addressed:

- [ ] Inconvenience acknowledged as a risk (never "no risks")
- [ ] Audio/video recording stated as "will be" not "may be", and transcription explained
- [ ] Location stated for every activity
- [ ] Time commitment includes PLS reading time
- [ ] Activities list in PLS matches consent form bullet points exactly
- [ ] Technical terms explained in plain language on first use
- [ ] Abbreviations spelled out on first use (in application AND participant documents)
- [ ] All external personnel have organisation details
- [ ] Withdrawal rights consistent with data type (anonymous = can't withdraw data)
- [ ] Consent form says "will be" for recording (not "may be")
- [ ] Consent form footer: Project ID + Project Start Date + Version
- [ ] PLS footer: Project ID + version number + date
- [ ] Dates in documents are not in the past
- [ ] Recruitment materials in plain language, with contact email and time commitment
- [ ] Study materials (vignettes, surveys, interview guides) attached or described
- [ ] If workshops: anonymity/confidentiality/reputation risks addressed
- [ ] If face-to-face: researcher safety risks addressed
- [ ] If multi-phase: participant count explicitly totalled across groups
- [ ] Location checkboxes consistent with actual research environments
- [ ] Risk-benefit justification addresses ALL identified risks
