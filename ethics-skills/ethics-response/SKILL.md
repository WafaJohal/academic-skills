# Ethics Response — Committee Review Reply

You will help draft a formal, point-by-point response to ethics reviewer feedback,
suitable for upload to Infonetica ERM as a "Response to review documents" file.

## Before you begin

Read `references/common-issues.md` — this contains common patterns of reviewer feedback
with standard fixes for each. When drafting responses, check whether the committee's
comment matches a known pattern and apply the standard resolution.

---

## What you need from the researcher

Before drafting, gather:
1. **The reviewer feedback** — paste the full committee/reviewer comments (can be a table, numbered list, or free text letter)
2. **The original application text** for the sections being queried — you need to know what was said before so you can explain the change
3. **Their proposed resolutions** — what do they want to change or clarify? Or do they want help deciding?
4. **Reference number** — the ethics application ID (for the document header)

Ask: "Have you already decided how you'll address each point, or would you like me to suggest responses first?"

---

## Before drafting: triage the feedback

Read all comments and categorise each:

| Category | What it means | Approach |
|----------|---------------|----------|
| **Information request** | Committee wants clarification or more detail | Add/expand in application; explain in response |
| **Genuine revision required** | Something in the design or documentation must change | Make the change; describe it precisely in response |
| **Misunderstanding** | Committee has misread the application | Politely clarify; point to existing text; expand if unclear |
| **Condition of approval** | Must be met before approval granted | Address directly; confirm it's been done |

Flag to the researcher if any comment involves a **design change** (e.g., changing consent approach, changing recruitment method) vs. a **documentation clarification** — because design changes may require more thought.

---

## Drafting Principles

1. **Address every single comment** — do not skip any, even if it seems minor
2. **Number responses to match the reviewer's numbering** — makes it easy for the committee to track
3. **State what was changed, not just that you agree** — e.g., "We have updated Section G, paragraph 2 to read: '[new text]'"
4. **Be respectful and non-defensive** — even if a comment reflects a misreading
5. **Quote the updated application text** where relevant — the committee should not have to hunt for changes in the application
6. **Track changes in application documents** — note that revised PLS/consent forms should have changes highlighted or tracked
7. **Be specific about where in the form changes were made** — "Section D, Background" or "Section G, Consent and information – all participant groups"

## Response Document Structure

```
RESPONSE TO REVIEWER COMMENTS

Project Title: [Title]
Ethics Reference Number: [ID]
Project Supervisor: [Name]
Date of Response: [Date]
Version: [e.g., Response v2]

---

We thank the [HREC / Low Risk Review Delegate] for their careful review of our application.
We have addressed all comments below and have revised the application and attached documents
accordingly. All changes to documents are highlighted [in yellow / with tracked changes].

---

COMMENT 1: [Paste or summarise the reviewer's exact comment]

RESPONSE 1:
[Your response. Include:]
[1. Acknowledgement of the concern]
[2. What change was made (or clarification provided)]
[3. Where in the application the change appears]
[4. The new/revised text if short enough to quote directly]

---

COMMENT 2: [...]

RESPONSE 2:
[...]

---

[Repeat for all comments]

---

SUMMARY OF CHANGES TO APPLICATION

The following sections of the application have been updated:
- Section [X]: [brief description of change]
- Section [Y]: [brief description of change]

The following attached documents have been revised:
- [Document name] (version [N] → version [N+1]): [brief description of changes]

We are happy to provide any additional information if required.
```

## Tone Guide

**When addressing a misunderstanding:**
> "We thank the reviewer for raising this point. To clarify, [restate the relevant information from the application]. We have expanded [Section X] to make this clearer: '[new text]'."

**When making a requested change:**
> "We agree with the reviewer's concern. We have [describe what was done]. The relevant section of the application has been updated to read: '[new text]'. The [PLS/Consent Form] has also been updated accordingly (see [Document name] v[N])."

**When defending a design decision:**
> "We appreciate the reviewer's attention to this aspect of the study. The [method/approach] was chosen because [rationale, citing NS or literature if applicable]. We have added additional explanation in [Section X] to make this rationale clearer: '[new text]'."

**Never:**
- Express frustration with the reviewer
- Simply say "We agree" without stating the actual change
- Ignore part of a multi-part comment
- Claim a change was made without quoting the new text

## Common Reviewer Comments and Standard Response Approaches

| Common comment | Standard approach |
|----------------|-------------------|
| "Sample size not justified" | Add statistical or methodological rationale; cite precedent for method (e.g., "consistent with participatory design norms: Muller & Kuhn, 1993") |
| "Consent form missing [element]" | Add the element; quote the added text; reference PLS/CF updated document |
| "Risk X not adequately addressed" | Expand risk section with specific mitigation; update PLS risk section too |
| "Unclear how data will be de-identified" | Add specific process: pseudonyms assigned, identifiers removed at [stage], stored separately |
| "Overseas ethics requirements not addressed" | Add country-specific response per NS Chapter 4.8; address GDPR if EU |
| "Conflict of interest not addressed" | Explicitly address relevant dimensions (autonomy, dependent relationships, dissemination restrictions) |
| "PLS too complex for general audience" | Simplify language; remove jargon; re-submit updated PLS |
| "Mandatory complaint paragraph missing" | Add exact institutional complaint paragraph with placeholder for ethics ID |
| "What happens to data after project ends?" | Specify: retention period (5 years), destruction method, who is responsible |
| "Recruitment via [third party] — privacy implications?" | Address APP/HPP compliance; confirm no identifiable data shared |

## After Drafting

Once the response document is drafted:

1. **List all application sections that need updating** — the researcher must update these in Infonetica before resubmitting
2. **List all attachment documents that need new versions** — PLS, consent forms, recruitment materials — remind researcher to highlight changes in revised versions
3. **Remind the researcher:** when resubmitting in Infonetica, they must:
   - Log in and address reviewer's comments directly in the form fields
   - Attach revised documents with new version numbers
   - Remove superseded document versions
   - Upload the response document to Section G under "Response to review documents"
   - Do NOT submit a new application — revise the existing one

4. **Check response document** against the original feedback to confirm every comment is addressed.
